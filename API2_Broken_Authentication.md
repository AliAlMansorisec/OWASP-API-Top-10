# 🌐 API2:2023 - Broken Authentication

---

## 📌 1. التعريف (Definition)

تحدث هذه الثغرة عندما تكون "بوابات الدخول" الخاصة بالـ API ضعيفة أو غير محمية بشكل جيد، مما يسمح للمخترقين بتوقع أو سرقة "مفاتيح الدخول" (Tokens/Passwords) الخاصة بالمستخدمين الآخرين.

في عالم الـ APIs، المصادقة (Authentication) هي العملية التي يتأكد فيها السيرفر أنك "أنت" فعلاً من خلال التحقق من:

- اسم المستخدم وكلمة المرور.
- أو مفاتيح الـ API (API Keys).
- أو رموز الوصول (Access Tokens مثل JWT).

**المشكلة تظهر عندما:** يستطيع المخترق تخمين هذه المفاتيح، أو تخطي خطوة التأكد، أو استغلال ضعف في كيفية توليد هذه المفاتيح أو تخزينها.

---

## 🧠 Real-World Analogy (مثل واقعي)

تخيل أنك في **نادي رياضي خاص** 🏋️‍♂️، وهذا النادي يستخدم **أجهزة ذكية** للدخول:

1. **المصادقة الصحيحة:** أنت تضع إصبعك على البصمة، الجهاز يتأكد أنك (علي)، ويفتح لك الباب. هذا نظام سليم.
    
2. **المشكلة (Broken Authentication):**
    
    - **تخمين الرمز:** تخيل أن الجهاز لا يستخدم البصمة، بل يطلب "كود" من 4 أرقام. أنت جلست وجربت `0000`, `0001`, `0002`.. والجهاز **لا يمنعك** من المحاولة! بعد قليل، ستجد الكود الصحيح وتدخل كأنك عضو في النادي.
        
    - **المفتاح المفقود:** أو تخيل أن النادي يعطي "بطاقة دخول"، وأنت وجدت بطاقة ملقاة على الأرض. الجهاز لا يسأل عن وجهك، هو فقط يقرأ البطاقة ويقول: "تفضل يا صاحب البطاقة".
        
    - **الباب الذي لا يغلق:** أو أنك دخلت خلف شخص آخر قبل أن يغلق الباب التلقائي، والجهاز لم يلاحظ أن هناك شخصين دخلا بمصادقة واحدة.

👉 **في عالم الـ API:**

- **الجهاز:** هو الـ API Endpoint.
    
- **الكود أو البطاقة:** هي الـ Access Token أو الـ API Key.
    
- **المخترق:** هو الشخص الذي يستغل أن السيرفر "سهل الخداع" ولا يضع قيوداً على عدد المحاولات، أو لا ينهي صلاحية المفاتيح القديمة.

---

## 🧩 2. أنواع الثغرة (Types of Broken Auth)

تتوزع هذه الثغرة على عدة أخطاء برمجية أو إدارية:

### 🔨 1. هجمات القوة الغاشمة (Brute Force Attacks)

أخطر أنواعها في الـ API هو عدم وجود "حد للمحاولات" (Rate Limiting).

- **المثال:** المخترق يرسل آلاف الطلبات لتخمين كلمة المرور أو الـ OTP (رمز التحقق) والسيرفر يظل يستقبل الطلبات دون أن يحظر الهكر.

### 🎫 2. رموز الوصول الضعيفة (Weak or Predictable Tokens)

أحياناً يتم توليد الـ Access Tokens بطريقة سهلة التوقع.

- **المثال:** استخدام توكن يعتمد على "اسم المستخدم + التاريخ" بتشفير بسيط مثل Base64. يمكن للمخترق "صناعة" توكن لأي مستخدم آخر والدخول به.

### 🔄 3. سوء إدارة الجلسات (Poor Session Management)

- **عدم انتهاء الصلاحية:** أن يظل الـ Token يعمل للأبد ولا تنتهي صلاحيته (No Expiration).
        
- **تسريب التوكن في الرابط:** إرسال الـ `access_token` في الرابط (URL) بدلاً من الـ `Headers`؛ مما يجعله يظهر في سجلات السيرفر (Server Logs) أو تاريخ المتصفح.

### 🔑 4. استخدام الـ Credentials غير الآمنة

- السماح باستخدام كلمات مرور ضعيفة جداً.
        
- عدم تغيير "كلمات المرور الافتراضية" للخدمات المرتبطة بالـ API (مثل قواعد البيانات أو لوحات التحكم).

### 📩 5. ضعف في استعادة كلمة المرور (Broken Password Reset)

- عندما يرسل الـ API رابط استعادة كلمة المرور، ويكون الرابط يحتوي على كود سهل التخمين أو يظل صالحاً حتى بعد استخدامه.

---

## 🔥 3. Impact (خطورة الثغرة)

تعتبر هذه الثغرة من "العيارات الثقيلة" لأنها تمنح المخترق المفاتيح الأساسية للنظام. إليك تفاصيل الأثر الناتج عنها:

- 👥 **الاستيلاء الكامل على الحسابات (Full Account Takeover):**
    
    هذا هو الأثر المباشر. بمجرد كسر المصادقة، يدخل المخترق إلى حساب أي مستخدم (أو حتى المدير Admin) ويتحكم فيه كأنه المالك الأصلي. يغير كلمة المرور، يسرق البيانات، أو يغلق الحساب.
    
- 🛡️ **انتحال الشخصية (Identity Theft):**
    
    في تطبيقات الـ API التي تتعامل مع البيانات المالية أو الطبية، يمكن للمخترق القيام بعمليات "باسم الضحية". (مثلاً: تحويل أموال من حساب بنكي عبر API البنك، والبنك سيظن أن العملية شرعية لأنها تمت بتوكن صحيح).
    
- 🔓 **الوصول لبيانات الشركة الحساسة (Unauthorized Access):**
    
    بما أن الـ APIs هي جسور بين الأنظمة، فإن كسر المصادقة قد يكشف "أسرار العمل" (Business Secrets)، خطط الشركة، بيانات الموظفين، أو الوصول لقواعد بيانات داخلية كانت معزولة خلف هذا الـ API.
    
- 📉 **انهيار السمعة والثقة (Reputational Loss):**
    
    إذا اكتشف العملاء أن نظام المصادقة ضعيف، ستفقد ثقتهم فوراً. لا أحد يريد أن يتعامل مع منصة يمكن لأي مخترق بسيط أن "يخمن" فيها طريق الدخول.
    
- 🔨 **التلاعب الشامل بالبيانات (Data Integrity Breach):**
    
    المخترق لن يكتفي بالمشاهدة؛ يمكنه حذف بيانات، تعديل سجلات، أو حتى زرع "أبواب خلفية" (Backdoors) لضمان الدخول في المرات القادمة حتى لو تم إصلاح الثغرة.

## 🛠️ 4. Discovery (كيف تكتشفها؟)

لاكتشاف فشل المصادقة، عليك فحص كل نقطة يتم فيها تبادل "الهوية" بين المستخدم والسيرفر. استخدم أدواتك مثل (Burp Suite) لتنفيذ الخطوات التالية:

### 🛑 1. اختبار حواجز الحماية (Rate Limiting Test)

هذه هي الخطوة الأولى والأهم في الـ APIs:

- **الطريقة:** اذهب إلى الـ `Login Endpoint` أو `Forgot Password`. حاول إرسال طلبات تسجيل دخول خاطئة متتالية بسرعة كبيرة باستخدام `Burp Intruder`.
    
- **ماذا تبحث عنه؟** إذا استمر السيرفر في الرد بـ `401 Unauthorized` لآلاف المرات دون أن يحظر الـ IP الخاص بك أو يطلب `Captcha` أو يمنع المحاولات مؤقتاً، فأنت أمام ثغرة **Broken Authentication**.

### 🎫 2. فحص الـ JWT والـ Tokens (Token Analysis)

ابحث عن:

- **ضعف التشفير:** انسخ الـ Token إلى موقع `jwt.io`. هل المعلومات (Payload) تحتوي على بيانات حساسة؟ هل يمكنك تغيير الـ `alg` (الخوارزمية) إلى `none`؟
    
- **التنبؤ بالتوكن:** هل التوكن يتغير بشكل كبير في كل مرة؟ أم أن هناك أجزاء ثابتة يمكن تخمينها؟
    
- **انتهاء الصلاحية:** سجل خروجك (Logout) ثم حاول استخدام الـ Token القديم مرة أخرى. إذا قبل السيرفر الطلب، فهذا يعني أن الجلسة لا تنتهي برمجياً.

### 🔑 3. فحص روابط "استعادة كلمة المرور" (Password Reset Flow)

- **التلاعب بالطلب:** عند طلب إعادة تعيين كلمة المرور، هل يمكنك تغيير الإيميل في الطلب (Body) ليرسل الرابط إلى إيميلك أنت بدل إيميل الضحية؟
    
- **تخمين الكود:** إذا كان الموقع يرسل كود (OTP) من 4 أرقام، هل يمكنك عمل `Brute Force` عليه وتخمينه خلال دقائق؟

### 🔍 4. فحص الـ API Keys في الـ Front-end

- ابحث في ملفات الـ `Main.js` أو الـ `Chunks` عن كلمات مثل `apiKey`, `secret`, `token`.
    
- أحياناً يترك المطورون مفاتيح الـ API (مثل مفاتيح Firebase أو Stripe) مكشوفة في كود المتصفح، مما يسمح لأي شخص باستخدامها.

### 🎭 5. اختبار "انتحال الشخصية" (Credential Stuffing)

- استخدم قائمة بكلمات مرور مسربة (Leaked Credentials) وجربها على الـ API. الـ APIs غالباً لا تملك أنظمة حماية متطورة ضد الـ Bot attacks مثل واجهات الويب التقليدية.

---

## 🛡️ 5. Remediation (كيف تُصلح؟)

لإغلاق ثغرة **Broken Authentication**، يجب اتباع نهج "الدفاع العميق" (Defense in Depth). إليك الخطوات مرتبة حسب الأهمية:

### 🛑 1. تطبيق تقييد المعدل (Implement Rate Limiting)

هذا هو الدفاع الأول ضد هجمات الـ Brute Force.

- **الإجراء:** حدد عدد محاولات تسجيل الدخول المسموح بها (مثلاً 5 محاولات كل 15 دقيقة) لكل عنوان IP أو لكل حساب مستخدم.
    
- **النتيجة:** سيضطر المخترق للانتظار سنوات لتخمين كلمة مرور واحدة، مما يجعل الهجوم غير مجدٍ.

### 📱 2. فرض المصادقة متعددة العوامل (Enforce MFA/2FA)

- **الإجراء:** لا تعتمد على كلمة المرور فقط. اطلب رمزاً من تطبيق (مثل Google Authenticator) أو رسالة نصية (رغم أن التطبيقات آمن).
    
- **النتيجة:** حتى لو سرق المخترق كلمة المرور، لن يستطيع الدخول بدون الجهاز الفيزيائي للمستخدم.

### 🎫 3. تأمين الرموز (Secure Token Management)

بما أن الـ APIs تعتمد على الـ JWT، يجب عليك:

- **Short-lived Tokens:** اجعل عمر الـ `Access Token` قصيراً جداً (مثلاً 15 دقيقة).
    
- **Refresh Tokens:** استخدم `Refresh Token` مؤمن لتجديد الجلسة.
    
- **Secure Storage:** لا تخزن التوكنز في `Local Storage` إذا كان ذلك ممكناً؛ استخدم `HttpOnly Cookies` لمنع سرقتها عبر ثغرات XSS.
    
- **Revocation List:** قم ببناء نظام لإبطال التوكن (Blacklist) عند عمل Logout.

### 🔑 4. تقوية سياسة كلمات المرور (Strong Password Policy)

- **الإجراء:** ارفض كلمات المرور الضعيفة (مثل 123456). استخدم مكتبات مثل `zxcvbn` لتقييم قوة كلمة المرور أثناء التسجيل.
    
- **الحماية:** لا تخزن كلمات المرور كنص واضح؛ استخدم خوارزميات Hash قوية مثل **Argon2** أو **Bcrypt**.

### 🕵️‍♂️ 5. منع تسريب المعلومات (Credential Leak Protection)

- **الإجراء:** استخدم رسائل خطأ غامضة. بدل أن تقول "كلمة المرور خاطئة" (مما يؤكد للهكر أن الإيميل صحيح)، قل "اسم المستخدم أو كلمة المرور غير صحيحة".
    
- **التحقق:** تحقق من قواعد بيانات التسريبات المعروفة (مثل API الخاص بـ _Have I Been Pwned_) لمنع المستخدمين من استخدام كلمات مرور مسربة مسبقاً.

### 🔄 6. حماية روابط استعادة الحساب (Secure Password Reset)

- **الإجراء:** استخدم توكنز عشوائية تماماً (Cryptographically Secure) لروابط الاستعادة، واجعلها صالحة للاستخدام **مرة واحدة فقط** وتنتهي خلال مدة قصيرة (مثلاً 10 دقائق).

## 🧠 6. Attacker Mindset (اسأل نفسك كمخترق)

عندما تقف أمام "بوابة الدخول" (Login Page/Endpoint)، اسأل نفسك هذه الأسئلة الاستفزازية للنظام:

- 🕵️ **هل هناك "باب خلفي" للمطورين؟**
    
    "هل ترك المبرمج حساباً تجريبياً (Test Account) بكلمة مرور سهلة مثل `admin/admin` أو `test/test`؟ هل توجد Endpoints غير موثقة مثل `/api/v1/debug/login` تسمح بالدخول بدون قيود؟"
    
- 🛡️ **هل الحارس "ينام" بعد عدد معين من الضربات؟**
    
    "إذا جربت 10 كلمات مرور خاطئة، هل سيحظرني السيرفر؟ وماذا لو غيرت الـ IP الخاص بي؟ ماذا لو جربت كلمة مرور واحدة على 100 حساب مختلف (Password Spraying) بدلاً من تجربة 100 كلمة على حساب واحد؟ هل سيكتشفني النظام؟"
    
- 🎫 **هل "تذكرة الدخول" سهلة التزوير؟**
    
    "عندما أنظر إلى الـ JWT الخاص بي، هل يبدو عشوائياً حقاً؟ ماذا لو حذفت التوقيع (Signature) وأرسلت الطلب؟ ماذا لو غيرت تاريخ الانتهاء لعام 2030، هل سيقبل السيرفر التذكرة؟"
    
- 🔄 **هل يمكنني "سرقة" جلسة شخص آخر؟**
    
    "إذا كان الموقع يرسل التوكن في الـ URL، هل يمكنني العثور عليه في الـ Google Analytics أو الـ Logs؟ إذا سجلت الخروج، هل تظل 'التذكرة' التي معي تعمل؟ إذا نعم، فأنا أملك مفتاحاً لا ينتهي مفعوله!"
    
- 📩 **هل نظام "نسيت كلمة المرور" ضعيف؟**
    
    "عندما يطلب النظام إيميلي لإرسال رابط الاستعادة، هل يمكنني اعتراض الطلب وتغيير الإيميل لإيميل مختبري؟ هل الكود المرسل (OTP) قصير بما يكفي لأقوم بتخمينه في ثوانٍ؟"
    
- 📱 **هل نسخة الموبايل "أكثر تساهلاً"؟**
    
    "غالباً ما يضع المطورون حماية قوية على موقع الويب، لكنهم ينسون الـ API الخاص بتطبيق الأندرويد أو الـ iOS. هل يطلب تطبيق الموبايل نفس قوة كلمة المرور؟ هل فيه Rate Limiting؟"
