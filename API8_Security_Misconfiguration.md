# ⚙️ API8:2023 — Security Misconfiguration

---

### 📌 1. التعريف (Definition)

ثغرة **Security Misconfiguration** (أو سوء الإعدادات الأمنية) هي مظلة واسعة تشمل أي خلل ناتج عن عدم تطبيق معايير الأمان عند إعداد السيرفر، أو قاعدة البيانات، أو الـ API نفسه.

### 🧠 Real-World Analogy (المثل الواقعي)

تخيل أنك استأجرت **خزنة متطورة جداً** في بنك عالمي، مصنوعة من أقوى أنواع الفولاذ وبها نظام تشفير معقد لا يمكن كسره.

1. **الوضع الآمن:** أن تقوم بتغيير الرمز السري الافتراضي، وتفعيل خاصية بصمة الوجه، والتأكد من إغلاق باب الخزنة جيداً بعد كل استخدام.
    
2. **المشكلة (ثغرة Security Misconfiguration):** تحدث الثغرة في إحدى الحالات التالية:
    
    - **ترك المفتاح في الباب:** أن تترك الخزنة مغلقة لكنك نسيت المفتاح بداخل القفل "سهواً".
        
    - **الرمز الافتراضي:** أن تترك الرقم السري كما جاء من المصنع (مثلاً `0000`) لأنك لم تكلف نفسك عناء تغييره.
        
    - **الباب الموارب:** أن تغلق الخزنة ولكن تترك باب "غرفة الصيانة" الخلفي مفتوحاً دون حراسة.
        
    - **لوحة الإعلانات:** أن تضع ملصقاً بجانب الخزنة يشرح بالتفصيل نوع القفل، وكيفية عمل النظام، ومن هو صانعها (وهذا يشبه **رسائل الخطأ المفصلة** التي تعطي المخترق معلومات تقنية).
        
3. **النتيجة:** يأتي السارق، ولا يحتاج لاستخدام أدوات كسر أو تفجير؛ هو فقط يجرب "الاحتمالات البديهية" أو يدخل من الباب الذي نسيته مفتوحاً، ويأخذ كل ما في الخزنة بكل هدوء.
    

👉 **في عالم الـ API:**

- **الخزنة:** هي الـ API الخاص بك وقاعدة البيانات.
    
- **المفتاح الافتراضي:** هو كلمة مرور `admin` أو `root` التي لم يتم تغييرها بعد التثبيت.
    
- **الباب الموارب:** هو منفذ (Port) مثل `21` (FTP) أو لوحة تحكم قاعدة البيانات المتروكة متاحة للجمهور.
    
- **لوحة الإعلانات:** هي رسالة الخطأ التي تقول للمخترق: _"فشل الاتصال بقاعدة بيانات MySQL إصدار 8.0.21 على السيرفر الداخلي 10.0.0.5"_. (هذه الرسالة كنز للمخترق!).

### 🧩 2. أنواع سوء الإعدادات (Types of Misconfiguration)

#### 1. كشف المعلومات التقنية (Verbose Error Messages) 📢

هذا النوع يحدث عندما يكون الـ API "ثرثاراً" أكثر من اللازم.

- **المشكلة:** عند حدوث خطأ (مثل إدخال بيانات خاطئة)، يعيد الـ API تفاصيل مثل (Stack Trace)، إصدارات المكتبات المستخدمة، أو هيكلية قاعدة البيانات.
    
- **لماذا هو نوع من سوء الإعداد؟** لأن وضع "التنقيح" (Debug Mode) تُرك مفاعلاً في بيئة الإنتاج (Production)، مما يعطي المخترق خريطة دقيقة لنقاط الضعف.
    

#### 2. الخدمات والميزات غير الضرورية (Unnecessary Features) 🛠️

غالبًا ما تأتي السيرفرات أو أطر العمل (Frameworks) بميزات مفعلة افتراضيًا لتسهيل حياة المطور، لكنها تشكل خطراً أمنياً.

- **أمثلة:**
    
    - ترك صفحات "المساعدة" أو "الاختبار" (Test Pages) متاحة.
        
    - ترك بروتوكولات قديمة مفعلة (مثل HTTP بدل HTTPS).
        
    - تفعيل ميزات إضافية في الـ HTTP Methods مثل `TRACE` أو `PUT` في أماكن لا تحتاجها.
        

#### 3. الإعدادات الافتراضية الضعيفة (Default Configurations) 🔑

هذا هو النوع الأكثر شيوعاً وخطورة.

- **المشكلة:** استخدام كلمات المرور التي تأتي مع النظام (مثل `admin/password` للوحة التحكم).
    
- **إعدادات الأمان المفقودة:** عدم ضبط "رؤوس الأمان" (Security Headers) مثل:
    
    - **CORS:** السماح لأي موقع في الإنترنت بطلب بيانات من الـ API الخاص بك.
        
    - **CSP:** غياب سياسة حماية المحتوى التي تمنع حقن السكربتات.
        

#### 4. الأذونات المفتوحة وسوء إدارة الصلاحيات (Permissive Configurations) 🔓

تتعلق بكيفية تواصل أجزاء النظام مع بعضها.

- **المشكلة:**
    
    - ترك حاويات التخزين السحابي (مثل S3 Buckets) عامة (Public) للجميع.
        
    - عدم تقييد الوصول إلى واجهة الـ API (Swagger/OpenAPI documentation) مما يسمح للمخترق بفهم كل مسارات النظام بسهولة.
        
    - ترك ملفات الإعدادات (`.env` أو `.git`) مكشوفة على السيرفر.

### 🔥 3. Impact (خطورة الثغرة)

#### 1. كشف أسرار النظام (Information Disclosure) 📖

عندما تُترك ميزة "التنقيح" (Debug Mode) مفعلة أو تكون رسائل الخطأ مفصلة:

- **الأثر:** يحصل المخترق على أسماء الجداول في قاعدة البيانات، مسارات الملفات على السيرفر، وإصدارات البرامج المستخدمة. هذه المعلومات تُسهل عليه اختيار "سلاح الاختراق" (Exploit) المناسب بدقة 100%.
    

#### 2. السيطرة الكاملة على الحسابات (Account Takeover) 🔑

إذا لم يتم تغيير الإعدادات الافتراضية للوحات التحكم:

- **الأثر:** يستطيع المخترق الدخول بكلمات مرور بديهية (Default Credentials) إلى أدوات الإدارة مثل _Admin Panel_ أو _Database Manager_. بمجرد الدخول، يمكنه تعديل بيانات المستخدمين، سرقة الإيميلات، أو تغيير صلاحيات أي حساب.
    

#### 3. سرقة البيانات الضخمة (Data Breaches) 📂

ترك حاويات التخزين السحابي (مثل S3 Buckets) أو قواعد البيانات (مثل MongoDB) مفتوحة للعامة دون كلمة مرور:

- **الأثر:** تسريب ملايين السجلات (صور، ملفات شخصية، بيانات دفع) في دقائق معدودة. هذا لا يسبب خسارة تقنية فقط، بل يؤدي إلى غرامات قانونية ضخمة وفقدان تام للسمعة.
    

#### 4. تعطيل الخدمة (Denial of Service - DoS) 🛑

الإعدادات التي تفتقر لقيود الموارد (Resources Limits):

- **الأثر:** يمكن للمخترق استغلال ميزات "ثقيلة" في السيرفر (مثل ميزة معالجة الصور التي تركت مفعلة) وإغراقها بالطلبات حتى يتوقف السيرفر عن العمل تماماً للمستخدمين الحقيقيين.
    

#### 5. كسر سياسة المنشأ (CORS Misconfiguration) 🌐

إذا تم ضبط `Access-Control-Allow-Origin` على النجمة (`*`) بشكل خاطئ:

- **الأثر:** يمكن لأي موقع خبيث في الإنترنت أن يسرق بيانات المستخدمين الحساسة (مثل الـ Tokens) من الـ API الخاص بك بمجرد أن يزور المستخدم ذلك الموقع الخبيث.



### 🛠️ 4. Discovery (كيف تكتشفها؟)

#### 1. فحص رسائل الخطأ (Error Probing) ⚠️

حاول إرسال بيانات "مستفزة" للـ API لإجباره على إظهار خطأ.

- **التجربة:** أرسل رموزاً غريبة مثل (`'`) أو (`"`) أو قيماً غير متوقعة في حقول الأرقام.
    
- **ماذا تبحث عنه:** هل تظهر رسالة خطأ تقنية (Stack Trace)؟ هل يظهر اسم قاعدة البيانات أو إصدار سيرفر الويب (مثل `nginx/1.18.0`)؟ أي معلومة تقنية تظهر هي مؤشر لسوء الإعداد.
    

#### 2. تجربة "البديهيات" (Default Credentials Testing) 🔑

عندما تجد لوحة تحكم أو صفحة تسجيل دخول لإحدى الخدمات الملحقة بالـ API.

- **التجربة:** جرب التركيبات الشهيرة: (`admin/admin`)، (`admin/password`)، (`guest/guest`).
    
- **الأدوات:** يمكنك استخدام أدوات مثل **Burp Suite Intruder** مع قائمة بكلمات المرور الافتراضية الشائعة.
    

#### 3. فحص ملفات التعريف والمسارات (Sensitive Files Hunting) 📂

ابحث عن الملفات التي تُترك عادةً عن طريق الخطأ أثناء عملية الرفع (Deployment).

- **التجربة:** حاول الوصول للمسارات التالية يدوياً:
    
    - `/.env` (يحتوي على مفاتيح الـ API وكلمات سر قاعدة البيانات).
        
    - `/.git/` (قد يسمح للمخترق بتحميل كود المصدر بالكامل).
        
    - `/swagger.json` أو `/v1/api-docs` (لفهم جميع مسارات الـ API).
        

#### 4. فحص رؤوس الأمان (Security Headers Check) 🛡️

استخدم المتصفح أو أدوات الفحص لرؤية الـ Headers في استجابة الـ API.

- **ماذا تبحث عنه:** غياب رؤوس مثل `X-Content-Type-Options: nosniff` أو `Strict-Transport-Security`.
    
- **فحص CORS:** جرب إرسال طلب بـ Header اسمه `Origin: http://evil.com`. إذا رد السيرفر بـ `Access-Control-Allow-Origin: http://evil.com` أو `*` فهذا سوء إعداد خطير.
    

#### 5. فحص المنافذ والخدمات (Port Scanning) 📡

استخدم أداة مثل **Nmap** لفحص السيرفر المستضيف للـ API.

- **التجربة:** هل هناك منافذ مفتوحة غير ضرورية؟ (مثل منفذ 21 للـ FTP، أو 3306 للـ MySQL متاح للعامة، أو منافذ لوحات التحكم مثل 8080).
    

#### 6. فحص الحاويات السحابية (Cloud Buckets) ☁️

إذا كان التطبيق يستخدم روابط لتخزين الملفات.

- **التجربة:** حاول تغيير اسم الملف في الرابط أو محاولة "سرد" (List) محتويات المجلد السحابي (مثل S3 Bucket). إذا استطعت رؤية قائمة الملفات، فهذا سوء إعداد في الأذونات.


### 🛡️ 5. Remediation (كيف تُصلح؟)

#### 1. أتمتة التقسية (Hardening Automation) 🤖

بدلاً من ضبط السيرفر يدوياً في كل مرة، استخدم أدوات "البنية التحتية ككود" (Infrastructure as Code).

- **الإجراء:** استخدم ملفات إعدادات جاهزة ومؤمنة (مثل Docker images أو Terraform scripts) لضمان أن كل سيرفر جديد يولد بإعدادات أمنية مثالية تلقائياً.
    
- **القاعدة:** لا تترك مجالاً للخطأ البشري في الإعدادات اليدوية.
    

#### 2. تعطيل وضع التنقيح (Disable Debug Mode) 🛑

هذا هو الإجراء الأهم عند نقل الكود من مرحلة التطوير إلى مرحلة الإنتاج (Production).

- **الإجراء:** تأكد من إغلاق أي ميزة تعيد تفاصيل تقنية أو "Stack Trace".
    
- **البديل:** اجعل الـ API يعيد رسائل عامة للمستخدم (مثل "حدث خطأ داخلي، يرجى المحاولة لاحقاً")، بينما يتم تسجيل التفاصيل التقنية في **ملفات سجلات (Logs) داخلية** مشفرة ومحمية للرجوع إليها من قبل المطورين فقط.
    

#### 3. تغيير كل ما هو "افتراضي" (Change Defaults) 🔑

- **الإجراء:** بمجرد تثبيت أي خدمة (قاعدة بيانات، سيرفر، لوحة تحكم)، قم فوراً بـ:
    
    - تغيير اسم المستخدم وكلمة المرور الافتراضية.
        
    - تغيير المنافذ (Ports) الافتراضية إن أمكن (مثلاً تغيير منفذ الـ SSH من 22 إلى رقم آخر عشوائي).
        
    - حذف الحسابات التجريبية (Guest/Test accounts).
        

#### 4. سياسة "الحد الأدنى" (Minimalism Policy) ✂️

- **الإجراء:** قم بإيقاف أي ميزة أو خدمة لا يحتاجها الـ API فعلياً.
    
    - أغلق المنافذ غير المستخدمة في الـ Firewall.
        
    - احذف الملفات غير الضرورية من السيرفر (مثل `.git` أو `.env.bak` أو ملفات التوثيق غير المحمية).
        
    - قيد الـ HTTP Methods المسموحة (مثلاً إذا كان الـ API للقراءة فقط، اسمح بـ `GET` وامنع `POST` أو `DELETE`).
        

#### 5. ضبط رؤوس الأمان (Security Headers) 🛡️

اجعل المتصفح والسيرفر يتواصلان بصرامة أمنية.

- **الإجراء:** أضف الرؤوس التالية في كل استجابة (Response):
    
    - `Content-Security-Policy`: لمنع حقن السكربتات.
        
    - `X-Content-Type-Options: nosniff`: لمنع المتصفح من تخمين نوع الملفات.
        
    - `Strict-Transport-Security` (HSTS): لإجبار الاتصال عبر HTTPS فقط.
        
    - **ضبط CORS:** لا تستخدم النجمة (`*`)؛ حدد فقط النطاقات الموثوقة التي يسمح لها بطلب البيانات.
        

#### 6. مراجعة دورية للأذونات (Permission Auditing) 🔍

- **الإجراء:** تأكد من أن حاويات التخزين (Cloud Buckets) وقواعد البيانات ليست "Public". استخدم مبدأ "أقل الصلاحيات" (Principle of Least Privilege)؛ أي أن كل خدمة تملك فقط الصلاحيات التي تحتاجها لتعمل، ولا تملك صلاحيات "Root" أو "Admin" بلا داعٍ.


### 🧠 6. Attacker Mindset (اسأل نفسك كمخترق)

#### 1. "هل المطور نسي 'مخلفات' عملية التطوير؟" 🧹

"الموقع يعمل بشكل ممتاز، لكن ماذا لو جربت الدخول إلى `/.env`؟ ربما نسوا حذف ملف الإعدادات الذي يحتوي على مفاتيح AWS أو كلمة سر قاعدة البيانات. ماذا عن مجلد `/.git`؟ إذا وجدته، سأقوم بتحميل كود المصدر بالكامل إلى جهازي وأبحث عن الثغرات في الكود براحتي."

#### 2. "هل السيرفر 'ثرثار' ويحب المساعدة؟" 🗣️

"سأقوم بإرسال طلب يحتوي على خطأ مقصود، مثل وضع حرف في حقل يتطلب رقماً. هل سيعطيني السيرفر رسالة خطأ بسيطة؟ أم سيقوم بعرض (Stack Trace) يخبرني أن التطبيق يستخدم `Node.js v14.15.0` ومكتبة `Sequelize`؟ هذه المعلومات تخبرني بالضبط ما هي الثغرات العامة (CVEs) التي يمكنني تجربتها ضد هذه الإصدارات."

#### 3. "ماذا لو كانت الأبواب الخلفية مفتوحة؟" 🚪

"المطور أمن المنفذ 80 و 443 (الويب). لكن هل نسي منفذ 3306 الخاص بقاعدة البيانات متاحاً للإنترنت؟ سأقوم بعمل Scan سريع للمنافذ. إذا وجدت منفذ قاعدة البيانات أو منفذ Redis مفتوحاً، سأحاول الدخول بكلمات المرور الافتراضية. قد يكون المطور قد افترض أن 'لا أحد سيعرف بوجود هذا المنفذ'."

#### 4. "هل يثق الـ API بأي شخص في الإنترنت؟" 🌐

"سأقوم بفحص إعدادات الـ CORS. إذا وجدت `Access-Control-Allow-Origin: *` فهذا يعني أنني أستطيع بناء موقع خبيث، وإذا زاره أي مستخدم مسجل دخول في هذا الـ API، يمكنني استخدام متصفحه لسرقة بياناته من الـ API دون علمه."

#### 5. "هل هناك خدمات منسية بلا حراسة؟" 🏗️

"كثير من المطورين ينسون واجهة **Swagger UI** أو **Postman Collections** متاحة للعامة. إذا وجدتها، سأعرف جميع الـ Endpoints، والمعاملات التي تقبلها، بل وحتى الأمثلة على البيانات. لقد وفروا عليّ نصف مجهود البحث (Reconnaissance)!"
