# 🌐 API3:2023 - Broken Object Property Level Authorization (BOPLA)

---

## 📌 1. التعريف (Definition)

تحدث هذه الثغرة عندما يسمح الـ API للمستخدم بالوصول إلى "خصائص" (Properties/Fields) معينة داخل الكائن لا ينبغي له الوصول إليها، سواءً كان ذلك **بالقراءة** أو **بالتعديل**.

ببساطة: الـ API قد يسمح لك برؤية ملفك الشخصي (هذا صحيح)، ولكن المشكلة تبدأ عندما يعطيك الـ API في استجابة الـ JSON حقولاً إضافية مثل `"is_admin": true` أو `"wallet_balance": 1000` (هذا تسريب بيانات)، أو عندما يسمح لك بإرسال طلب تعديل لبروفايلك وتغيير حقل `"role": "admin"` بنجاح (هذا حقن خصائص).

---

## 🧠 Real-World Analogy (المثل الواقعي)

تخيل أنك تعمل في **شركة كبيرة**، ولكل موظف **بطاقة تعريفية (ID Card)** معلقة على صدره:

1. **الوضع الطبيعي:** البطاقة تظهر (اسمك، صورتك، مسمّاك الوظيفي). هذا أمر عادي والجميع مسموح له برؤية هذه البيانات.
    
2. **المشكلة (BOPLA - جانب القراءة):** تخيل لو أنك قلبت البطاقة، ووجدت مكتوباً في خلفها بخط صغير (الراتب الشهري، الرقم السري لخزنة الشركة، والعنوان المنزلي للموظف).
    
    - **في الـ API:** السيرفر أرسل لك "البطاقة"، لكنه أرسل معها بيانات "خلفية" حساسة لم يطلبها أحد، والمبرمج اعتمد على أن الناس لن "تقلب" البطاقة وتنظر للخلف!
        
3. **المشكلة (BOPLA - جانب التعديل):** تخيل أنك ذهبت لمكتب الشؤون الإدارية لتغيير "صورتك الشخصية" فقط. الموظف أعطاك استمارة فيها (الاسم، الصورة، الراتب، الرتبة). أنت قمت بتغيير الصورة، وبالمصادفة قمت بشطب راتبك وكتبت **"10,000$"** وشطبت مسمى "موظف" وكتبت **"مدير عام"**. الموظف أخذ منك الورقة وأدخلها في الكمبيوتر **دون أن يتأكد** هل يحق لك تعديل خانة الراتب والرتبة أم لا، وقام بتحديث بياناتك فوراً!
    
    - **في الـ API:** هذا ما نسميه **Mass Assignment**؛ السيرفر استقبل منك "كل البيانات" وحدّثها في قاعدة البيانات دون فلترة.

## 🧩 2. أنواع الثغرة (BOPLA Types)

تنقسم هذه الثغرة إلى نمطين أساسيين، وكلاهما يعتمد على "عدم وجود فلترة" للخصائص (Properties):

### النوع الأول: تسريب الخصائص الحساسة (Excessive Data Exposure)

هذا النوع يحدث في طلبات القراءة (**GET Requests**). المبرمج يقوم بجلب الكائن بالكامل من قاعدة البيانات ويرسله كما هو، معتمداً على أن الواجهة الأمامية (Front-End) ستعرض فقط "الاسم والصورة".

- **المثال:** تطلب بيانات حسابك عبر `/api/user/me`.
    
- **الاستجابة المسربة:**
    
```json
{
  "username": "ali_mansoori",
  "avatar": "url...",
  "is_admin": false, // خاصية حساسة لا يجب أن تظهر
  "internal_notes": "User is a pentester", // ملاحظات داخلية
  "password_hash": "$2b$12$..." // كارثة: تسريب الهاش
}
```

### النوع الثاني: التعيين الجماعي أو حقن الخصائص (Mass Assignment)

هذا النوع يحدث في طلبات الكتابة أو التعديل (**POST/PUT/PATCH**). المبرمج يستقبل البيانات من المستخدم ويقوم بحفظها مباشرة في قاعدة البيانات (Binding) دون تحديد الحقول المسموح بتعديلها.

- **المثال:** تريد تعديل اسمك فقط. ترسل طلب `PATCH /api/user/update`.
    
- **الهجوم:** تقوم بإضافة حقول لم يطلبها النموذج (Form) ولكن السيرفر "يقبلها":

```json
{
  "username": "ali_new_name",
  "is_admin": true, // حقن خاصية لترقية الصلاحيات
  "balance": 999999 // حقن خاصية لزيادة الرصيد
}
```

---

## 💡 لماذا دمجتهما OWASP في ثغرة واحدة (BOPLA)؟

لأن العلة واحدة: **"ضعف التحكم في مستوى الخصائص"**.

- في الأول: السيرفر لا يتحكم فيما "يخرج" من الخصائص.
    
- في الثاني: السيرفر لا يتحكم فيما "يدخل" من الخصائص.

## 🔥 3. Impact (خطورة الثغرة)

تتنوع المخاطر هنا بناءً على نوع البيانات التي يتعامل معها الـ API، ولكنها غالباً ما تقع تحت هذه الكوارث الثلاث:

### 1. ترقية الصلاحيات (Privilege Escalation) 🚀

هذا هو الخطر الأكثر شيوعاً. يستطيع المستخدم العادي تحويل نفسه إلى "مدير" (Admin) أو "مشرف" (Moderator) ببساطة عن طريق حقن خصائص مثل `role` أو `is_admin`.

- **الأثر:** السيطرة الكاملة على لوحة التحكم، حذف مستخدمين آخرين، أو سرقة بيانات النظام بالكامل.
    

### 2. التلاعب المالي (Financial Loss) 💰

في تطبيقات التجارة الإلكترونية أو المحافظ الرقمية، يمكن التلاعب بحقول الأسعار أو الأرصدة.

- **الأثر:** تغيير سعر منتج من 1000$ إلى 1$ في طلب الـ `POST` (Mass Assignment)، أو زيادة رصيد المحفظة عبر حقن حقل `balance`.
    

### 3. تسريب الأسرار التقنية (Information Disclosure) 🕵️‍♂️

عندما يسرب الـ API حقولاً داخلية (Internal Fields) مثل مفاتيح الـ API لخدمات أخرى، أو مسارات الملفات في السيرفر، أو حتى الـ `Password Hashes`.

- **الأثر:** هذا التسلل للمعلومات يمهد الطريق لهجمات أكبر (مثل كسر الهاشات أو اختراق خدمات الطرف الثالث المرتبطة بالتطبيق).
    

### 4. تدمير منطق العمل (Business Logic Breach) 🏗️

تخيل تطبيقاً يسمح للمستخدم بتغيير حالته من "طالب" إلى "خريج" ليحصل على ميزات معينة، أو تغيير تاريخ اشتراكه ليجعله "صالحاً للأبد".

- **الأثر:** خسائر مادية فادحة للشركة وتخريب لقاعدة البيانات التي تصبح مليئة بالمعلومات المغلوطة.

## 🛠️ 4. Discovery (كيف تكتشفها؟)

كمختبر اختراق، هدفك هو العثور على حقول (Fields) لم تظهر في واجهة المستخدم، أو محاولة "تخمين" حقول إدارية وتمريرها للسيرفر.

### 1. اكتشاف تسريب البيانات (Data Exposure) 🔍

- **راقب الـ JSON الخام:** استخدم **Burp Suite** لمراقبة استجابات السيرفر (Responses). قارن بين ما يظهر على الشاشة (Front-end) وبين الـ JSON الكامل القادم من السيرفر.
    
- **ابحث عن حقول غريبة:** ابحث عن كلمات مثل `role`, `admin`, `permissions`, `internal`, `status`, `balance`, `created_at`.
    
- **جرب Endpoints مختلفة:** أحياناً `/api/user/123` يعطي بيانات عادية، لكن `/api/admin/user/123` (إذا استطعت الوصول إليه) قد يسرب خصائص أكثر عن نفس المستخدم.
    

### 2. اكتشاف حقن الخصائص (Mass Assignment) 💉

هنا تحاول "إجبار" السيرفر على قبول بيانات لم يطلبها:

- **استراتيجية التخمين (Fuzzing):** عند إرسال طلب `PUT` أو `PATCH` لتحديث بياناتك، قم بإضافة حقول شائعة لم تكن موجودة في الطلب الأصلي.
    
    - _مثال:_ إذا كان الطلب الأصلي يرسل `{"name": "ali"}`, جرب إرسال `{"name": "ali", "is_admin": true}`.
        
- **البحث في الـ Documentation:** إذا كان هناك ملف `Swagger` أو `Postman Collection` متاحاً، ابحث عن جميع الخصائص الممكنة للكائن (Schema). قد تجد خصائص "للمدراء فقط" يحاول المبرمج إخفاءها برمجياً لكن الـ API يقبلها.
    
- **استخدام الـ OPTIONS Method:** أرسل طلب `OPTIONS` للـ Endpoint لترى ما هي الـ Methods المسموحة، وأحياناً السيرفر يسرب معلومات عن الـ Schema المتوقعة.
    
### 3. مقارنة الاستجابات (Diffing) ⚖️

- قم بإنشاء حسابين (مستخدم عادي ومستخدم بصلاحيات أعلى إن أمكن).
    
- لاحظ الفرق في أسماء الحقول (Properties) في استجابة كل منهما.
    
- خذ الحقول الموجودة في حساب "المدير" وحاول إرسالها (Injection) من حساب "المستخدم العادي".

## 🛡️ 5. Remediation (كيف تُصلح؟)

لإغلاق ثغرة **BOPLA** بشكل نهائي، يجب أن تتبع هذه الإجراءات الصارمة في الكود الخاص بك:

### 1. استخدام "القائمة البيضاء" (Whitelisting / Pick) ✅

هذه هي الطريقة الأكثر أماناً لمواجهة الـ **Mass Assignment**. لا تسمح للسيرفر بأخذ كائن كامل وتخزينه مباشرة.

- **بدلاً من:** `User.update(req.body)` (هذا خطر جداً).
    
- **الصحيح:** حدد الحقول المسموح بتعديلها فقط.

```js
// مثال باستخدام مكتبة lodash أو الاختيار اليدوي
const safeData = {
  name: req.body.name,
  bio: req.body.bio
  // لا نضع حقل is_admin هنا أبداً
};
User.update(safeData);
```

### 2. استخدام الـ DTOs (Data Transfer Objects) 🏗️

في الأنظمة الكبيرة، نستخدم كائنات وسيطة (DTOs) لتعريف شكل البيانات التي "تدخل" والتي "تخرج".

- **Input DTO:** يحدد فقط الحقول التي يُسمح للمستخدم بإرسالها.
    
- **Output DTO:** يحدد فقط الحقول التي يُسمح للمستخدم برؤيتها (وهذا يمنع تسريب البيانات الحساسة مثل الـ Passwords أو الـ Roles).
    

### 3. تجنب الوظائف التلقائية (Avoid Auto-Binding) 🚫

تجنب الوظائف التي تقوم بربط مدخلات المستخدم (JSON) مباشرة بنماذج قاعدة البيانات (Database Models) دون فلترة وسيطة. هذه الميزة مريحة للمطورين لكنها "جنة" للمخترقين.

### 4. فلترة المخرجات (Response Filtering) 🛡️

لا تعتمد على الـ Front-end لإخفاء البيانات. يجب أن يتم تجريد الكائن من الخصائص الحساسة قبل أن يغادر السيرفر.

### 5. التحقق من الصلاحيات لكل خاصية (Property-Level Auth) 🔐

إذا كان لا بد من وجود حقل يمكن تعديله من قبل "المدير" فقط (مثل حقل `status` لطلب معين)، يجب إضافة شرط برمجي داخل الكود:

- "إذا كان الحقل المراد تعديله هو `status` والمستخدم الحالي ليس `admin` -> **ارفض الطلب فوراً**".

## 🧠 6. Attacker Mindset (اسأل نفسك كمخترق)

عندما تتعامل مع أي API Endpoint، وتحديداً في طلبات البروفايل أو الإعدادات، تقمص دور "المحقق" واسأل النظام هذه الأسئلة:

- **🕵️ "ماذا لو أرسلتُ أكثر مما طُلِب مني؟"**  
"الموقع يطلب مني تغيير (اسمي) فقط. ماذا لو أضفتُ في الطلب `,"role": "admin"`؟ هل سيتم تجاهلها بذكاء، أم أن السيرفر سيأخذها بـ 'نية طيبة' ويحدث صلاحياتي؟"
    
- **🔍 "هل هناك بيانات 'للعيون التقنية' فقط؟"**  
"عندما أستلم استجابة الـ JSON، هل هناك حقول لم أرها في صفحة الويب؟ إذا وجدت حقلاً مثل `internal_id` أو `subscription_status`.. هل يمكنني محاولة إرساله في طلب التعديل (PATCH) لتغيير حالتي؟"
    
- **🧩 "كيف يبدو شكل قاعدة البيانات في عقل المبرمج؟"**  
"إذا كان هذا تطبيق متجر، فمن المؤكد أن جدول المنتجات يحتوي على حقل للسعر (`price`) وحقل للخصم (`discount`). حتى لو لم يظهر حقل الخصم في واجهة تعديل المنتج، سأحاول حقنه يدوياً في الطلب!"
    
- **🎭 "هل يمكنني 'تخمين' أسماء الحقول المخفية؟"**  
"المبرمجون غالباً ما يستخدمون أسماء نمطية. سأجرب إضافة حقول مثل: `is_verified`, `is_vip`, `account_balance`, `permissions`. ربما أحدهم يصيب الهدف!"
    
- **🛠️ "هل السيرفر 'كسول' ويستخدم Mass Assignment؟"**  
"سأقوم بمراقبة طلب الـ POST عند التسجيل. إذا كان يرسل 5 حقول، سأحاول إرسال 10. إذا لم يعطني السيرفر خطأ (Error) وقبل الطلب، فهذا يعني أنه غالباً يمرر البيانات مباشرة لقاعدة البيانات دون فلترة (Blackbox testing)."
